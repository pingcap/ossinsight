Last week I noticed something weird in the GitHub trending data. A repo created on March 6th had 54,000 stars by March 25th.

That's 19 days. That's [karpathy/autoresearch](https://ossinsight.io/analyze/karpathy/autoresearch).

For context: [karpathy/nanoGPT](https://ossinsight.io/analyze/karpathy/nanoGPT) — which basically triggered the small-model Renaissance — took three years to reach the same number. His follow-up, [karpathy/nanochat](https://ossinsight.io/analyze/karpathy/nanochat), got there in 160 days. autoresearch did it in 19.

Something has changed.

## What autoresearch Actually Does

The premise is almost absurdly simple: give an AI agent a real training setup, let it modify the code overnight, and see if the model gets better.

The entire repo is three files. `prepare.py` handles data and tokenizer — the agent is **forbidden from touching it**. `train.py` is a single-file GPT implementation with a sliding-window attention architecture, a custom MuonAdamW optimizer, and value embeddings — this is the *only* file the agent edits. And `program.md` is the specification that tells the agent the rules of the game.

Here's how the loop actually works, straight from the code:

1. The agent creates a git branch (`autoresearch/<tag>`), reads all three files for context, and runs `train.py` unmodified to establish a **baseline** `val_bpb` (bits per byte).
2. It proposes a change to `train.py` — architecture, optimizer, hyperparameters, batch size, model depth, anything. It commits the change.
3. It runs `uv run train.py > run.log 2>&1`. Training runs for a **fixed 5-minute wall-clock budget**, then evaluates on a pinned validation shard.
4. It checks the single metric: `val_bpb`. Lower is better.
5. If improved → **keep the commit**, advance the branch. If equal or worse → **`git reset`** back to the previous state.
6. Log results to a TSV. Go to step 2. **Never stop.**

That last part is literal. From `program.md`: *"Do NOT pause to ask the human if you should continue. The human might be asleep. You are autonomous. The loop runs until the human interrupts you, period."*

At ~5 minutes per experiment, that's ~12 experiments per hour, ~100 overnight. You wake up to a `results.tsv` with a hundred attempted improvements, the successful ones auto-committed on the branch.

At its core, this is not an agent in the way most people use the word. It's a **search algorithm with git as its state machine.** The agent asks exactly one question after each run: *is this number lower?* No planning across experiments, no memory of what worked three hours ago (beyond what's in the current `train.py`), no meta-strategy. Just propose → commit → train → measure → keep or revert → repeat.

This simplicity is also its boundary. The loop only works when three conditions hold: a **clear scalar metric** (`val_bpb` — computed by a fixed evaluation function the agent cannot modify), a **fast feedback cycle** (5-minute training budget on a single GPU), and **safe reversibility** (`git reset` on failure). There's even a simplicity criterion baked in: *"A 0.001 val_bpb improvement that adds 20 lines of hacky code? Probably not worth it. A 0.001 improvement from deleting code? Definitely keep."*

Most real research problems satisfy none of these conditions. The domains where "run it overnight" works are exactly the domains where the search space is tractable and the objective is well-defined. That's a powerful subset — but it's a subset.

Coding agents do your work. Research agents do your *thinking*.

## The Pattern Underneath

autoresearch isn't alone. When I looked at adjacent repos that gained significant traction in the last 60 days, a pattern emerged: the move from "AI helps you code" to "AI runs the loop without you":

| Repository | Stars | Created | What it does |
|-----------|-------|---------|-------------|
| [karpathy/autoresearch](https://ossinsight.io/analyze/karpathy/autoresearch) | 54,553 | Mar 6 | AI agents run ML experiments overnight |
| [jennyzzt/dgm](https://ossinsight.io/analyze/jennyzzt/dgm) | 1,944 | May 2025 | Darwin Gödel Machine: open-ended evolution of self-improving agents |
| [facebookresearch/HyperAgents](https://ossinsight.io/analyze/facebookresearch/HyperAgents) | 666 | Mar 19 | Self-referential agents that optimize for any computable task |
| [zerobootdev/zeroboot](https://ossinsight.io/analyze/zerobootdev/zeroboot) | 1,885 | Mar 2026 | Sub-millisecond VM sandboxes for AI agents to run safely |

None of these repos are about building better prompts or writing cleaner code. They're about taking humans *out of the loop* for repetitive cycles — experiment → measure → iterate — and replacing that loop with an agent that runs it 200 times while you eat dinner.

The progression is becoming visible:

1. **2023-2024**: AI as copilot (Copilot, Cursor, Aider)
2. **2025**: AI as autonomous coder (Claude Code, Codex, OpenCode)
3. **2026**: AI as autonomous researcher (autoresearch, DGM, HyperAgents)

Here's the stat I keep coming back to: autoresearch has **7,594 forks** but only **7 contributors** to the main repo.

That ratio — 1,085 forks per contributor — is unlike anything I've seen at this scale. For comparison:

What does a 1,085:1 fork-to-contributor ratio mean? People are taking the base, running it with *their own models*, their own `program.md`, their own compute — and not contributing back to main because there's nothing to contribute. The core is already done. The research happens in private forks.

This is a fundamentally different model from open-source software development. It's closer to open-source science: share the methodology, not the results.

## The Bottleneck That Didn't Move

I've seen a lot of celebration about autoresearch being "the future of ML research." Maybe. But there's a harder question buried in the design.

Look at the division of labor in the code. The human writes `program.md` — which defines the evaluation metric (`val_bpb`), the constraints (don't touch `prepare.py`, don't add dependencies), and the philosophy (simplicity over complexity). The human also writes the 300-line `prepare.py` that defines how data is loaded, how tokenization works, and the exact evaluation function. **The agent cannot modify any of this.** It operates *within* a sandbox the human designed.

The agent's autonomy is real but bounded: it can change the model architecture (transformer depth, attention patterns, embedding strategies), the optimizer (learning rates, momentum schedules, weight decay), the training loop — anything in `train.py`. But it cannot change what "better" means. It cannot decide to use a different dataset, a different evaluation metric, or a different time budget.

That's not a flaw — it's a deliberate and carefully engineered constraint. But it means the bottleneck hasn't moved as far as it looks. The scientist hasn't been automated. The *technician* has — a very good technician that never sleeps and never gets bored, but a technician nonetheless.

Most ML experiments are not bottlenecked by running training loops. They're bottlenecked by knowing which experiments are worth running — by the intuition that says "try value embeddings with a gated residual" instead of "try a bigger model." autoresearch removes the former but not the latter. The part that requires the 30 years of intuition, the domain expertise, the aesthetic sense about what's elegant — that part is still the human's job.

Which means this is Stage 1, not the endgame. autoresearch is the most primitive form of a larger pattern emerging across repos like DGM and HyperAgents: AI systems that don't just execute experiments, but run entire research cycles end-to-end — with persistent memory, evolving strategy, and eventually, self-generated hypotheses. The full stack looks something like: memory + loop + abstraction. autoresearch only has the loop. But the loop alone is already getting 54K stars in 19 days.

## What Comes Next

If autoresearch represents the current frontier, the next wave of repos will likely tackle what it deliberately left out:

1. **Automated hypothesis generation** — not just running experiments, but deciding *which* experiments to run based on prior results. [jennyzzt/dgm](https://ossinsight.io/analyze/jennyzzt/dgm) (1,944 stars) is already exploring this: agents that evolve their own objectives, not just optimize a fixed one.

2. **Cross-fork knowledge transfer** — right now, those 7,594 forks are islands. Each person's `program.md` produces insights that stay private. The first tool that aggregates experimental findings across forks — turning private research into collective learning — will unlock enormous value.

3. **Evaluation bootstrapping** — agents that improve their own benchmarks, not just their performance on fixed ones. [facebookresearch/HyperAgents](https://ossinsight.io/analyze/facebookresearch/HyperAgents) (666 stars, created March 19) is a week old and already exploring self-referential optimization.

These repos are small today. But autoresearch was small 20 days ago.

If there's one sentence to take away from this analysis: **research is becoming search.** Not in the Google sense — in the optimization sense. Define a metric, define a search space, let the loop run. The question is no longer whether AI can do research. It's how much of research is reducible to search — and what remains irreducibly human.
